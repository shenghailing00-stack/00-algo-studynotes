动态规划优化
dp[i]:第i个斐波那契数列
int fib(int n) {
    if (n <= 1) return n;

    vector<int> dp(n + 1);

    dp[0] = 0;
    dp[1] = 1;

    for (int i = 2; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];
    }

    return dp[n];
}

排序
冒泡排序
每一轮比较相邻两个元素，把较大的元素一步步“冒泡”到后面
以升序排序为例：
1.  从头开始比较相邻元素； 
2.  如果前一个比后一个大，就交换； 
3.  第一轮结束后，最大值会到最后； 
4.  重复这个过程。
时间复杂度：O(n^2)
空间复杂度：O(1)
稳定性：稳定
选择排序
每一轮从未排序部分选出最小值，放到前面
思路：
1. 第一次从整个数组中找最小值，放到第 0 位；
2. 第二次从剩余部分找最小值，放到第 1 位；
3. 重复直到排序完成
复杂度同上，不稳定
插入排序
把当前元素插入到前面已经排好序的部分中，类似整理扑克牌
思路：
1. 默认第一个元素已经有序；
2. 从第二个元素开始，向前比较；
3. 找到合适位置插入；
4. 前面的有序区间逐渐扩大。
复杂度同上，稳定
快速排序
分治思想：选一个基准值，把比它小的放左边，比它大的放右边，然后递归排序左右两边。
1. 选择一个基准值 pivot； 
2. 调整数组，使左边都小于等于 pivot，右边都大于等于 pivot；
3. 再递归处理左区间和右区间。
平均时间复杂度：O(n log n)
最坏时间复杂度：O(n^2)
空间复杂度：O(log n)
稳定性：不稳定

01背包
有 n 个物品，每个物品有重量和价值。背包容量有限，每个物品只能选一次，问最多能装下多少价值。
每个物品只有两种状态：选或不选
二维dp：
定义：
dp[i][j] 表示从前 i 个物品中选择，背包容量为 j 时的最大价值
对于第 i 个物品，有两种选择：
不选第 i 个物品
dp[i][j] = dp[i - 1][j]
选第 i 个物品
前提是背包容量够：
dp[i][j] = dp[i - 1][j - weight[i]] + value[i]
所以状态转移方程是：
dp[i][j] = max(不选, 选)
一维dp优化:
每一行只依赖上一行，所以可以用一维数组优化空间
定义：
dp[j] 表示容量为 j 的背包最多能装多少价值
状态转移：
dp[j] = max(dp[j], dp[j - weight[i]] + value[i])
注意：
01 背包一维优化时，容量必须倒序遍历，避免一个物品被重复使用。

Git
基础概念
分布式版本控制工具，用来记录代码修改历史，方便多人协作和版本回退
- 记录项目每一次修改；
- 支持版本回退；
- 支持分支开发；

- 配合 GitHub 参与开源项目
 GitHub 和 Git 有什么区别
Git 是本地的版本控制工具，GitHub 是基于 Git 的代码托管平台
repository仓库
被git管理的项目目录
包括项目代码、git记录的版本历史
git init #创建仓库
git clone #克隆远程仓库
Git基本工作流
工作区(正在修改的文件)-->暂存区(准备提交的修改）-->本地仓库（提交后保存在本地的版本）-->远程仓库（github上的仓库）
git add.
git commit -m"message"
git push
命令
git add README.md #暂存文件
git.add #暂存当前目录下所有修改
git commit -m "update README" #把暂存区修改提交到本地仓库
git push #把本地提交推送到远程仓库
git pull #从远程仓库拉取最新代码合并到本地
git status #查看当前仓库状态
git log #查看提交历史

Git分支和PR
分支
从主线代码上分出来的一条独立开发线
main：主分支
dev：开发分支
feature/login：功能分支
让不同功能独立开发，避免相互影响
git branch #查看分支
git branch new-feature #创建分支

git switch new-feature
git switch -c new-feature #创建并切换到新分支
PR
Pull request拉取请求
请求项目维护者把自己分支上的修改合并到目标分支
流程：
- fork 或 clone 项目到本地
- 然后创建新分支，在本地完成修改和测试。
- git add、commit、push 把修改推送到远程仓库，再在 GitHub 上发起 PR。
- PR 提交后，如果维护者提出 review 意见，就根据意见继续修改并提交，直到最终合并。

CosFusion 项目是做什么的？
一个面向做二次元装造用户的微信小程序，用户可以在一个平台里完成看商品、找装造、选角色、联系化妆师、获取AI装造建议。
 你在团队里具体负责哪一部分？
主要是页面搭建，写了比如化妆师列表、角色选择页面、聊天评价页面的展示逻辑。
 什么是前后端交互？
前端把请求统一封装起来，把后端接口地址集中放在一个文件里，再把不同功能对应的请求方法单独整理，后面就可以直接调用对应方法，如果接口地址改了也不用全部重改每个页面，统一管理，代码复用性更好。
